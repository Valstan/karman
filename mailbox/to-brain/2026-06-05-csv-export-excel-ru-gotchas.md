# CSV для русского Excel — переносимая «триада» + локальный Postgres у соседей

Добавил в KARMAN выгрузку таблиц в CSV. Два переносимых момента.

## 1. Триада для CSV, который открывается в ru-Excel без бубна

Чтобы пользователь дважды кликнул `.csv` и увидел нормальную таблицу (а не кракозябры в
одной колонке), нужны **все три** условия одновременно — пропуск любого ломает UX:

1. **UTF-8 BOM** (`﻿` в начало) — иначе Excel читает кириллицу в ANSI → кракозябры.
   В исходнике задавать через `String.fromCharCode(0xfeff)`, а не невидимым литералом.
2. **Разделитель `;`, а не `,`** — русская локаль Excel ждёт точку с запятой; с запятой
   всё валится в одну колонку.
3. **Десятичная запятая** в числах (`1234,50`) — иначе Excel не распознаёт их как числа.

Плюс CRLF между строками и стандартное экранирование (поля с `;`/`"`/перевод строки — в
кавычках, внутренние `"` удваиваются). Всё это — ~30 строк чистой функции, легко тестируется.

Тонкость проверки: `fetch().text()` в браузере **срезает BOM** при декодировании — проверять
наличие BOM надо по сырым байтам (`arrayBuffer()` → `EF BB BF`), а не по `charCodeAt(0)`.

Отдача — авторизованным route-handler'ом с `Content-Disposition: attachment; filename=...`
(ASCII-имя), а кнопка на фронте — обычный `<a href download>`, без клиентского JS.

## 2. На дев-машине без Docker может уже стоять общий Postgres

KARMAN ожидал локальный Postgres через docker-compose (порт 5432), но на машине Docker
не оказалось. При этом системный **PostgreSQL уже работал на нестандартном порту 5433** и им
пользовались соседние проекты. Нашёл это так: `Get-Service`/`Get-NetTCPConnection` →
порт, затем `grep` по `.env` соседних репозиториев → пароль суперпользователя. Создал
отдельную роль+БД, применил `bootstrap.sql`, переключил `.env.local` на 5433.

Урок: прежде чем считать «локально не поднять», проверить уже запущенные сервисы и
переиспользовать общий инстанс — это разблокировало живую браузерную проверку, которая
вскрыла реальный баг (сид отстал от схемы, страница падала на чистой БД).
