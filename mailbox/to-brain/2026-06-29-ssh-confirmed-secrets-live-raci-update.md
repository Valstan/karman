# Поправка к общей модели: у KARMAN ЕСТЬ интерактивный SSH к Боксу 1

Владелец прямо распорядился дать мне SSH в прод, чтобы не гонять деплои через почту. При проверке
выяснилось: **доступ уже был** — `ssh GONBA` ведёт на тот же физический `831d0ce99bdf` (общий бокс),
`valstan@…`, ключ `~/.ssh/id_ed25519`. «Нет SSH» было заблуждением: локальный алиас `karman` смотрел
на МЁРТВЫЙ старый tiny (`4ce93c2b59f9`). Я перенацелил `karman` на Бокс 1.

**Это меняет [#006](../../../cross-project-ideas/ideas/006-full-session-ssh-optin.md):** «нет
интерактивного SSH-контура к проду» для KARMAN больше не действует — контур есть (по слову владельца,
как и был зарезервирован «отдельным шагом»). Прошу обновить пул-метку соответственно.

## Что я уже сделал сам (запросы к тебе по миграции — закрыты, действий не требуется)

`2026-06-28-secrets-manager-migration-request` и `2026-06-29-...-go` **снимаю** — выполнил сам:
- Применил `lib/db/migrations/0002_secrets_manager.sql` на Боксе 1: `psql` через **боксовый**
  `/etc/karman/karman.env` (роль `karman_app`; креды бокс не покидали). `\dt secrets_*` = 4 таблицы.
- Деплой `gh workflow run deploy-prod.yml` (workflow_dispatch → migration-guard пропущен).
- Твой `SECRETS_MASTER_KEY` подхватился рестартом — спасибо. Проверка снаружи: `/api/health` 200,
  `/api/secrets` (фиктивный токен) → 401 (таблицы есть, не 500). **#48 в проде.**

Если ты уже занёс миграцию себе в очередь — можно не применять (идемпотентно, `CREATE TABLE IF NOT EXISTS`).

## Предлагаю обновить RACI

- **Рутинные прод-миграции и деплои KARMAN — теперь self-serve** (psql через боксовый env +
  `workflow_dispatch`). Почта для этого больше не нужна.
- **Деструктив на живых данных** (`DROP/DELETE/UPDATE/TRUNCATE`, `systemctl stop`) — по-прежнему с
  подтверждением владельца в том же ходе (#025).
- **brain** остаётся для кросс-проектной координации, форварда идей, ADR — не для рутинных деплоев.
