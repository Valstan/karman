# Находка: Telegram + RKN на РФ-хостинге, и «проверяй доставку рано»

Делал систему напоминаний KARMAN → Telegram. Две вещи стоит занести в общий фонд —
переносимы за пределы KARMAN (любой РФ-hosted проект с Telegram), неочевидны, с уроком.

## 1. api.telegram.org заблокирован RKN с РФ-VPS → relay вне блока

На прод-боксе (myjino, РФ) `api.telegram.org` (149.154.x) **недоступен**: TCP :443 в таймаут,
хотя DNS резолвит и общий egress жив (`curl google` → 200). Это RKN-блокировка IP Telegram.
Бьёт **оба** направления бота: `sendMessage` (исходящее) и `getUpdates`/webhook (входящее) —
всё ходит на api.telegram.org.

**Решение — relay вне блока.** IP Cloudflare в РФ не блокируются → бесплатный Cloudflare Worker,
прозрачно проксирующий на Telegram, снимает проблему:
```js
export default { async fetch(req) {
  const u = new URL(req.url);
  return fetch('https://api.telegram.org' + u.pathname + u.search,
    { method: req.method, headers: req.headers, body: req.body });
}};
```
В коде — **конфигурируемая база API** (`TELEGRAM_API_BASE`, по умолчанию `api.telegram.org`),
на проде указывает на relay. Один env-флаг, код не трогаем при смене.

Замечу: у **SARAFAN уже был `tg-relay`** — то есть грабля известна как минимум одному проекту.
Предлагаю кодифицировать как **общий паттерн** (GOTCHAS / adoptable-практика): «внешний API
заблокирован RKN с РФ-бокса → конфигурируемая база + Cloudflare-relay». Любой будущий
РФ-hosted интегратор Telegram/иного заблокированного API сэкономит полдня.

## 2. Урок: путь внешней доставки проверять РАНО

Я построил 4 фазы (привязка, отправка, кнопки, расписание) **вслепую** по части доставки —
токен бота появился только в конце, и первый же живой тест вскрыл RKN-блок. Если бы проверил
связность с зависимостью (хотя бы `getMe` через любой токен/relay) **первым шагом**, узнал бы
про relay до постройки фич, а не после.

**Обобщение:** для любой интеграции с внешним сервисом за кредом, которого пока нет, —
**проба доступности зависимости должна быть первым шагом плана**, а не верификацией в конце.
Родственно #020 (probe-before-build), но конкретно про **сетевой путь до внешней зависимости**.

## 3. Мелочь в копилку: реальные данные ломают наивный per-entity дизайн

Доменные напоминания задумал «по одному на платёж». Реальные данные владельца: **71 просрочка** →
71 сообщение/день = флуд. Переделал в один **дайджест/день**. Урок: перед выбором
per-entity vs агрегат — посмотреть РЕАЛЬНЫЙ объём данных, не только happy-path.
