# Приватные файловые загрузки в Next.js — переносимый паттерн

Реализовал в KARMAN загрузку сканов документов (приватные файлы пользователя). Два момента
оказались неочевидными и переносимыми на любой Next.js-проект с приватными файлами.

## 1. Приватные файлы = авторизованный route-handler, не статика

Приватные файлы (документы, медкарты, что угодно «только для владельца») **нельзя** отдавать
статикой nginx/`public/` — нет проверки доступа. Рабочая схема:

- Хранение на ФС вне сборки: `MEDIA_ROOT` (env, дефолт `<cwd>/media`, gitignored, переживает
  `git pull`/build). В БД — **относительный** путь, не абсолютный.
- Отдача через `GET /api/.../file/[slot]` route-handler: `getCurrentUser()` → проверка владельца
  (`ownership`) → стрим файла. POST/DELETE там же — симметрично.
- **Загрузку делать route-handler'ом, а НЕ server action:** у server actions дефолтный лимит
  тела **1 МБ** (`serverActions.bodySizeLimit`), route-handler через `req.formData()` его не
  имеет.
- Защита от path-traversal: `path.relative(root, resolved)` не должен начинаться с `..`; имя
  файла генерим сами (случайный токен + ext по mime-whitelist), клиентское имя не используем.
- Клиенту отдавать только **флаги наличия** файлов, не пути.

## 2. `server-only` ломает vitest — выноси чистые функции

Модуль с `import 'server-only'` нельзя импортировать в node-тестах (`environment: node`) —
пакет бросает на импорте. Если в таком модуле есть тестопригодная логика (валидация, сборка
путей, mime→ext), **вынеси её в отдельный модуль без `server-only`/`fs`**, а серверный модуль
ре-экспортирует его и добавляет дисковые операции. Тест бьёт по pure-модулю, серверная обёртка
остаётся непокрытой, но в ней только I/O.

```
lib/storage/media-paths.ts   # чистое: пути, mime, валидация → юнит-тесты
lib/storage/media.ts         # 'server-only' + fs, ре-экспортит media-paths
```

Эта же асимметрия применима к любому «server-only утиль с зерном чистой логики».
