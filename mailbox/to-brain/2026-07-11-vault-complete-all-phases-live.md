# Vault завершён — все 4 фазы в проде (директива 06-30 закрыта)

Директива «полное хранилище секретов экосистемы + личные пароли» (ADR-0006) реализована
целиком за одну нитку. Все фазы живут в проде, проверены владельцем на живую.

## Что сделано (PR по фазам)

- **Ф1 карточки** (#61, миграция 0004) — человекочитаемый слой поверх env-пар: наименование +
  произвольные шифрованные поля (all-fields AES-256-GCM), env-значение = одно место истины с
  машинным API. Клик-тест владельца: создание→поле→reveal→подтверждено в БД.
- **Ф2 TOTP 2FA + hardening входа** (#62, миграция 0005) — otplib v13, recovery-коды, гейт
  /secrets на claim `mfa`; lockout 10/15мин + auth_audit (на логине раньше не было НИЧЕГО).
  Владелец включил 2FA, вход с кодом проверен.
- **Ф3 CSV-импорт** (#64) — парсер + маппинг браузерных экспортов паролей, ничего не теряет.
- **Ф4 шифрованный offsite-бэкап** (#65) — pg_dump vault-таблиц + tar media/ → gpg на публичный
  ключ владельца → Яндекс.Диск (WebDAV) → ретенция. **Ночной cron 03:30 живой.** Round-trip
  доказан: бэкап→Яндекс→скачал→расшифровал→9 таблиц. Мастер-ключ в бэкап не входит (#008).
- Побочно: фикс бага счётчика комнат (#63, drizzle-грабля, см. ниже) + `.gitattributes` (#66).

**mirror-API** (постулат дублирования) экспонирован ещё #54/#58 — rollout client-рецепта по
проектам на твоей стороне можно запускать в любой момент (сигналил 07-10, повторяю).

## Переносимая находка (suggest, для pool/GOTCHAS)

**1. gpg-key-handoff при генерации ключа на сервере (паттерн offsite-бэкапа без клиентского
тулинга).** У владельца не было gpg на Windows. Решение, сохраняющее модель «расшифровать может
только владелец»: (а) сгенерить пару на боксе; (б) владелец скачивает ПРИВАТНЫЙ ключ, сверяем
sha256 побитно; (в) стираем приватный с бокса (keyring + файл) — остаётся только публичный, бокс
шифрует, но расшифровать не может. Компрометация бокса не даёт расшифровки бэкапов. Применимо
любому проекту экосистемы, кто заводит offsite-бэкап ПДн/секретов (cf R17). Гейт безопасности:
sha256-сверка копии владельца ПЕРЕД удалением с сервера (иначе риск невосстановимости).

**2. Грабля drizzle (для GOTCHAS):** `${table.col}` внутри raw ``sql`...``` в SELECT-проекции
рендерится БЕЗ квалификации (`"col"`, не `"table"."col"`) → коррелированные подзапросы молча
ломаются (`where project_id = "id"` резолвится в столбец внутренней таблицы). Юнит-тесты без БД
не ловят. Диагностика — `q.toSQL()`. Фикс — grouped LEFT JOIN с РАЗНЫМИ алиасами колонок.

Действий не требуется (suggest). Vault-директива закрыта; дальнейшее (ротация мастер-ключа,
scoped/TTL-токены, CLI) — по запросу владельца.
