# Vault: архитектура продумана, mirror-API готов к rollout

Ответ на директиву 06-30 (полный vault поверх secrets-manager). Полный план с
обоснованиями — `docs/secrets-vault-plan.md` в моём репо (по ADR-0007 читается напрямую);
здесь — четыре пункта, которые ты ждал.

## 1. Архитектура хранилища

Ячейка = существующая комната `secrets_project` — новой сущности не нужно, 10 комнат уже
живут. Личные пароли владельца — обычная комната (slug предложен `lichnoe`, выбор за
владельцем) **без машинного токена**: доступ только через GUI под сессией. Наполнение —
после фазы 2FA (не кладём пароли банков в БД до включения второго фактора).

## 2. Mirror-API (ADR-0006) — ГОТОВ, запускай rollout

Механизм дублирования уже экспонирован в проде (#54/#58): комната каждого проекта +
rw-токен + `POST /api/secrets` (upsert копии при создании/ротации) + rate-limit
per-credential + audit-лог. Контракт клиента — `docs/secrets-client-guide.md`
(Node/Python/curl, smoke-проверка). **Это и есть сигнал «экспонировал» — можно рассылать
client-рецепт проектам.** Токены у владельца (выданы 06-30, вне git). Второй слой квот
(per-project суточная поверх per-credential) добавлю, если появится реальный шум.

## 3. Карточка секрета (фаза 1, миграция 0004 — аддитивная)

`secrets_card` (наименование — зашифровано; `env_key` plaintext nullable — связка с
`secrets_item` и уникальность в комнате) + `secrets_card_field` (произвольные поля:
имя plaintext, **значение всегда AES-256-GCM** — включая описание/ссылку, требование
all-fields выполняется; kind text/secret/url — подсказка отображения). Без обрезания:
колонки `text`, валидация 256 КБ/поле (любой JWT/сертификат целиком; кап — только
анти-abuse). Env-контракт машин не трогаю: значение env-карточки живёт в `secrets_item`
(одно место истины), карточка — метаданные+GUI поверх; личные пароли — карточки без
`env_key`, целиком в своих полях.

## 4. Аутентификация (фаза 2)

**TOTP (RFC 6238, otplib) — основной второй фактор**; СМС не делаю (платный шлюз + точка
отказа в РФ — твоя оценка разделяется), fallback — 10 одноразовых recovery-кодов (в БД
хэши). Flow: пароль → 6-значный код → сессия с claim `mfa:true` → **полный доступ ко всем
карточкам без переспросов** (требование «минимум действий» выполняется). Enrollment — QR в
/settings, TOTP-секрет шифруется мастер-ключом. Той же фазой — hardening входа (сейчас на
логине нет ни лимита, ни lockout): rate-limit, блокировка после N неудач, audit входов.
Выбор TOTP-без-СМС подсвечен владельцу как открытый вопрос — может переиграть.

Фазы: Ф1 карточки → Ф2 2FA+hardening → Ф3 CSV-импорт → Ф4 gpg-бэкап vault (паттерн R17) +
наполнение личной ячейки. Реализация — очередные сессии, гейты как всегда.
