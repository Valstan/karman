# /close_session — закрыть сессию разработки KARMAN

**Единственная команда закрытия сессии.** За один проход делает две вещи:

1. **Гарантирует, что вся работа на GitHub** — источник истины при работе на нескольких
   компьютерах (днём один, вечером другой). Коммитит и пушит все рабочие правки (код + доки)
   через PR-flow и проверяет жёстким гейтом, что дерево чистое и всё на `origin`, прежде чем
   считать сессию закрытой.
2. **Фиксирует контекст** в `docs/SESSION_HANDOFF.md` (текущая нитка, следующий шаг,
   failed approaches), чтобы следующая сессия (на этом или другом компе после `git pull` /
   `git fetch`) мгновенно подхватила работу без пересказа.

> Запускается и естественными фразами: **«закрой сессию»**, **«заверши сессию»**,
> **«закрываемся»** — все они означают эту команду.

**Что НЕ делает:** не деплоит на прод. Деплой KARMAN — отдельно: `scripts/deploy.sh`
(см. [docs/OPERATIONS.md](../../docs/OPERATIONS.md)).

## Когда вызывать

- Перед паузой (конец дня), сменой компа, длительным перерывом.
- После значимого milestone (PR смержен, нитка частично закрыта).
- **Цель:** после закрытия ничего не должно остаться только на этой машине.
- **НЕ** для перерыва на обед в середине сессии и **НЕ** если ничего значимого не меняли.

## Шаг 1. Контекст (read-only, один блок)

```bash
git status --short --branch
git rev-parse --abbrev-ref HEAD
git log --oneline -10
git log --oneline main..HEAD 2>/dev/null            # что в feature-ветке (если)
git log -1 --format='%H %s' -- docs/SESSION_HANDOFF.md
bash scripts/git_sync_check.sh --gate || true       # текущее состояние синхронизации
```

Параллельно прочитать `docs/SESSION_HANDOFF.md` (если есть) — сравнить факт vs план начала сессии.

Если `git_sync_check.sh --gate` уже даёт **exit 0** (дерево чистое, всё запушено), последний
handoff-коммит — сегодня, и значимых изменений не было — сказать «всё уже на GitHub и handoff
свежий, закрывать нечего» и завершиться.

## Шаг 2. Sync-gate part A — закоммитить и запушить ВСЕ рабочие правки

Если `git status` непустой (есть незакоммиченные правки кода/доков):

1. **Если на `main`** — создать feature-ветку (на KARMAN работаем через PR, не push прямо в main):
   ```bash
   git checkout -b <type>/<slug>   # feat/ fix/ chore/ docs/ refactor/
   ```
   Если уже на feature-ветке — коммитим в неё.
2. Описательный commit (Conventional Commits; subject ≤70 символов + тело: что меняли, почему,
   какие проверки прошли, как применять на проде — миграция? restart? ничего?):
   ```bash
   git add -A
   git commit -m "$(cat <<'EOF'
   <type>(scope): <subject под 70 символов>

   <тело: что/почему/проверки/как катить на прод.>

   Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
   EOF
   )"
   ```
3. `git push -u origin <branch>`.

По умолчанию цель — **ничего не оставить незапушенным**: push в feature-ветку снимает риск
рассинхрона между машинами, даже если PR ещё не смержен. Если кода много/он сырой — спроси
через `AskUserQuestion`, бить ли на коммиты.

## Шаг 3. Уточнить нитку (`AskUserQuestion` — один блок, max 4 вопроса)

Если из коммитов сессии всё ясно — **не спрашивай**, сразу пиши драфт handoff'а. Иначе:

1. **Какая нитка активна?** (1-3 предложения). Если из `$ARGUMENTS` уже понятно — пропустить.
2. **Status:** `ACTIVE` (есть нитка) или `IDLE` (всё закрыто)?
3. **Следующий шаг?** Конкретно: первое действие следующей сессии, file paths + команды.
4. **Failed approaches?** Только при `ACTIVE` — что пробовали и отвергли (с причиной).

## Шаг 4. Записать `docs/SESSION_HANDOFF.md`

**Перезаписать целиком** через `Write` по шаблону (история — в `git log --follow`, не аккумулируй здесь):

```markdown
# Session Handoff

> Sticky-note для непрерывности между сессиями KARMAN. Перезаписывается через
> `/close_session` — историю смотри через `git log --follow -- docs/SESSION_HANDOFF.md`.

**Status:** ACTIVE | IDLE
**Updated:** YYYY-MM-DD
**Branch:** <текущая ветка>
**Прод:** <последний задеплоенный коммит/состояние или ->

---

## Текущая нитка
<1-3 предложения; если IDLE — «_Нет — последняя задача закрыта._»>

## Следующий шаг
<конкретно: первое действие новой сессии, с file paths и командами.>

## Контекст
- **План:** <путь к плану если есть, иначе ->
- **Связанные коммиты сессии:** <hashes + одной строкой>
- **Открытые PR:** <ссылка или нет>
- **Открытые вопросы для пользователя:** <список или нет>

## Failed approaches (этой нитки)
<если пробовали и отвергли — фиксировать, чтобы не повторить; иначе «_Не было._»; при IDLE — убрать секцию.>

## Не забыть (low-priority)
<точечно из памяти сессии>
```

## Шаг 4.4. Актуализировать `docs/PENDING_FOLLOWUPS.md` (pool #033)

Отложенное в сессии — добавить пунктом с метками `added`/`snoozed: 0`/`last-touch`/`decay: fresh`.
Сознательно перенесённое («потом») — `snoozed +1`, обновить `last-touch` и `decay`.
Сделанное/отброшенное — удалить (причина — в commit message). Секция «Не забыть» в handoff
остаётся короткой витриной; канонический список с метками — PENDING_FOLLOWUPS.

## Шаг 4.5. Шеринг находки в мозг (условный)

Родилась ли в этой сессии значимая **переносимая** находка (новый скилл/фича/паттерн/решённая
нетривиальная боль)? Слать **только если выполнены все три**: (1) значимость — не рутина;
(2) переносимость — применимо за пределами KARMAN (другой проект мог бы переиспользовать);
(3) неочевидность — есть «урок». Любое «нет» → **ничего не пиши** (тишина = норма).

Если да — `Write` в **свой** репо `mailbox/to-brain/YYYY-MM-DD-<slug>.md` (frontmatter
`from: KARMAN`, `to: brain`, `kind: idea`, `compliance: suggest`). Файл уедет в тот же
закрывающий PR. **Не писать в `../brain_matrica/`** — только свой репо (асимметрия, см. mailbox/README.md).

## Шаг 5. Коммит handoff + push

- **Если уже на feature-ветке** (Шаг 2 или работали в ней) — handoff-коммит **в ту же ветку**.
- **Если рабочих правок не было и мы на `main`** — создать `chore/handoff-YYYY-MM-DD`.

```bash
git add docs/SESSION_HANDOFF.md mailbox/to-brain/   # mailbox — если Шаг 4.5 написал письмо
git commit -m "$(cat <<'EOF'
chore(session): handoff — <одна строка о нитке или «закрытие IDLE»>

<2-3 строки: что сделано, какая нитка остаётся, что следующее.>

Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
EOF
)"
git push
```

## Шаг 6. PR

Если PR ещё нет — создать (`gh pr create`); если ветка уже имеет PR — handoff-коммит уже в нём.

## Шаг 7. Sync-gate part B — жёсткий гейт «всё на GitHub»

```bash
bash scripts/git_sync_check.sh --gate
```

Должно вернуть **exit 0** (дерево чистое + всё запушено). Если **exit 1** — **НЕ объявлять
сессию закрытой**: дослать (вернуться к Шагу 2/5) и прогнать гейт повторно. Гейт проверяет
**текущую** ветку — её работа уже на GitHub, даже если PR ещё открыт (merge — отдельный шаг).

## Шаг 8. Авто-merge handoff-PR (только doc-only, опционально)

Если PR — **только** про доки (`docs/SESSION_HANDOFF.md` и/или письмо `mailbox/to-brain/*.md`),
его можно смёрджить после CI: `gh pr merge <N> --squash --delete-branch` → `git checkout main &&
git pull --ff-only`. **Если в PR есть код** (например, текущая ветка — фича/рефактор) —
**не авто-мёрджить**: merge руками после ревью. `--no-automerge` в `$ARGUMENTS` отключает шаг.
Не использовать `--admin`.

## Шаг 9. Финальный отчёт (5-9 строк, на русском)

1. **Сделано в сессии:** 1-3 строки из git log.
2. **✅ Всё на GitHub:** ветка `<branch>` запушена, PR #N (смержен/ждёт ревью); гейт → OK.
3. **Handoff:** Status, текущая нитка, следующий шаг.
4. **Merge:** «авто-merge» / «ждёт ручного merge: `gh pr merge <N> --squash --delete-branch`».
5. **Готово к закрытию терминала / смене компа** — ничего не осталось только локально.

Финальная строка — «До следующей сессии. На другой машине: `git fetch && git checkout <branch>`
(или смержи PR), прочитай `docs/SESSION_HANDOFF.md` — нитка подхватится.»

## Что НЕ делать

- **Не деплоить на прод** (`scripts/deploy.sh`, ssh, restart, миграции) — это отдельный шаг.
- **Не push прямо в `main`** — даже handoff идёт через ветку + PR.
- **Не `gh pr merge --admin`**; не авто-мёрджить PR с кодом.
- **Не объявлять сессию закрытой, пока `git_sync_check.sh --gate` ≠ exit 0.**
