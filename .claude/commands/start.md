# /start — открыть сессию разработки KARMAN

Парная команда к [`/close_session`](close_session.md). Подхватывает контекст без пересказа.

## Шаг 1. СНАЧАЛА синхронизация, ПОТОМ чтение состояния (pool #032)

Жёсткий порядок: handoff читается **только после** pull — иначе работаешь по устаревшему
состоянию, а свежий handoff с другой машины приедет позже и спутает картину.

```bash
git fetch
git status --short --branch          # дерево чистое?
git pull --ff-only                   # ТОЛЬКО при чистом дереве; конфликт/не-ff — разобраться руками
bash scripts/git_sync_check.sh --warn
git log --oneline -5
```

Грязное дерево или расхождение веток — **сначала** разобраться (это может быть оборванная
сессия — см. `/obriv`), и только потом продолжать.

## Шаг 2. Прочитать состояние (после sync!)

1. `docs/SESSION_HANDOFF.md` — нитка. При устаревшем `Updated:` (старше пары недель) — не
   доверять слепо, сверить с `git log`.
2. Почта brain: `cd ../brain_matrica && git pull --ff-only` (если там чистый main; чужую
   рабочую ветку не трогать) → новые письма в `mailboxes/KARMAN/from-brain/*.md`.
3. `docs/PENDING_FOLLOWUPS.md` — **всплыть** пункты с `decay: stale` (открыто > 30 дней или
   `snoozed ≥ 3`): по каждому ре-триаж тремя исходами — возобновить / переформулировать под
   текущий код / выкинуть с причиной (pool #033). Не всплывшее — не трогать.

## Шаг 3. Подсветить нитку

Кратко (3-6 строк):
- **Status / нитка** из handoff + **следующий шаг**.
- **Почта:** есть ли необработанные письма от brain (директивы — в очередь задач).
- **Отложенное:** всплывшие stale-пункты и итог ре-триажа.
- **Синхронизация:** предупреждения `git_sync_check.sh` — разобраться **до** новой работы.
- Если на другой машине осталась feature-ветка с открытым PR — напомнить
  `git fetch && git checkout <branch>`.

> Завершать сессию — через `/close_session` (закоммитит+запушит всё, обновит handoff, проверит гейт).
