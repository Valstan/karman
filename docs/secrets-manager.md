# Менеджер секретов

Зашифрованное хранилище секретных ключей проектов (как менеджер паролей) — модуль
KARMAN. Проекты получают свои ключи напрямую по токену, если потеряли локальный
экземпляр.

## Модель и угрозы (важно)

- Значения секретов хранятся **зашифрованными** (AES-256-GCM) в таблице `secrets_item`.
  Мастер-ключ (`SECRETS_MASTER_KEY`) лежит **только в окружении сервера**, не в БД и не в репо.
- Шифрование защищает от **утечки дампа БД** и от чтения значений в репозитории.
- Оно **НЕ защищает от компрометации самого сервера**: раз проекты получают plaintext
  по токену, сервер обязан уметь расшифровывать → имеет мастер-ключ в памяти. Это
  неизбежная цена авто-восстановления. Зеро-knowledge с авто-выдачей по токену несовместим.
- AAD шифра привязан к `secrets:<projectId>:<key>` — переставить шифротекст между строками БД нельзя.
- Токены доступа хранятся **только SHA-256-хэшем**; сам токен показывается владельцу один раз.

## Конфигурация

Мастер-ключ — base64 от 32 байт:

```bash
openssl rand -base64 32
```

Положить в окружение сервера (не в репо):

```
SECRETS_MASTER_KEY=<сгенерированный ключ>
```

Без ключа: страница `/secrets` позволяет создавать проекты, но шифрование и выдача
секретов отключены (`instrumentation.ts` пишет предупреждение в лог на старте,
`secretsConfigured()` → false, баннер в UI).

> ⚠️ Потеря мастер-ключа = невозможность расшифровать уже сохранённые секреты.
> Хранить ключ надёжно и отдельно от бэкапов БД.

## Миграция (прод — через brain)

Таблицы заводит `lib/db/migrations/0002_secrets_manager.sql` (рукописная, аддитивная).
На Боксе 1 у меня нет SSH → миграцию применяет brain через psql **до** деплоя
(migration-guard в `deploy-prod.yml` заблокирует авто-деплой при новой миграции —
деплой через `workflow_dispatch` после применения SQL). Локально:

```bash
psql "$DATABASE_URL" -f lib/db/migrations/0002_secrets_manager.sql
```

## Использование проектом (API)

```bash
# Все секреты проекта:
curl -H "Authorization: Bearer skm_…" https://<host>/api/secrets
# → {"secrets":{"DATABASE_URL":"…","API_KEY":"…"}}

# Один ключ:
curl -H "Authorization: Bearer skm_…" "https://<host>/api/secrets?key=DATABASE_URL"
# → {"secrets":{"DATABASE_URL":"…"}}  (или 404, если ключа нет)
```

Коды: `401` — нет/недействителен/отозван токен; `404` — нет запрошенного ключа;
`429` — rate-limit; `500` — мастер-ключ не настроен.

## Токены

- Создаются в UI проекта; показываются **один раз**. Метка (`name`) — чтобы отличать.
- Отзыв — `revoked_at`; отозванный токен сразу даёт 401.
- Каждое обращение пишется в `secrets_audit` (выдача/отказ/промах), обновляется `last_used_at`.

## Ограничения / на будущее

- **Rate-limit** — in-memory fixed window (60/мин на токен+IP), per-instance. На Боксе 1
  один инстанс — ок; при масштабировании заменить на общий стор (Redis). См. `lib/secrets/rate-limit.ts`.
- Доступ по токену — **read-only** (запись только через UI владельца).
- Ротация мастер-ключа (re-encrypt всех значений), envelope-ключи на проект, scoped/TTL-токены,
  экспорт/импорт `.env`, CLI-клиент — кандидаты следующих итераций.
