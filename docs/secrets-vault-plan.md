# Vault-план: полноценное хранилище секретов экосистемы + личные пароли

> Ответ на директиву brain 2026-06-30 (`mailboxes/KARMAN/from-brain/2026-06-30-karman-full-vault-spec.md`,
> ADR-0006). Расширение живого secrets-manager (#48/#54/#58, `docs/secrets-manager.md`) до
> полного vault: карточки секретов с произвольными полями, личные пароли владельца, 2FA,
> импорт, шифрованный бэкап. Статус реализации — по фазам ниже.

## Что уже есть (фундамент, в проде)

- **Ячейки** = `secrets_project` («комнаты»): 10 живых (9 экосистемных + trener), slug `[a-z0-9-]`.
- **Env-секреты** = `secrets_item` (`(project_id, key)` → AES-256-GCM, AAD привязывает
  шифротекст к строке, мастер-ключ `SECRETS_MASTER_KEY` вне БД/репо — pool #008).
- **Машинный API** = `GET/POST /api/secrets` (Bearer `skm_…`, в БД только SHA-256-хэш,
  `can_write` для записи, rate-limit 60/мин per credential+IP, audit-лог).
- **GUI владельца** = `/secrets` (комнаты, ключи, токены, аудит, reveal по клику).

## Решения

### 1. Ячейки: без изменений + личная ячейка владельца

Ячейка = существующая комната. Личные пароли владельца (почта, банки) — обычная комната
(slug предложен `lichnoe`, владелец может выбрать свой), просто без машинного токена:
доступ только через GUI под сессией. Никакой новой сущности не требуется.

### 2. Карточка секрета (фаза 1)

Новый слой поверх env-пар — **карточка** с человекочитаемыми метаданными и произвольными
полями. Env-контракт для машин не трогаем: `secrets_item` остаётся единственным хранилищем
env-значений (проекты продолжают push/pull как раньше), карточка — надстройка GUI.

Миграция `0004` (аддитивная, только новые таблицы):

- `secrets_card` — id, `project_id` (FK, cascade), `title` (наименование, зашифровано),
  `env_key` (программное обозначение; **plaintext**, nullable — у личных паролей его нет;
  уникально в пределах комнаты), timestamps.
- `secrets_card_field` — id, `card_id` (FK, cascade), `name` (имя поля: «Описание»,
  «Ссылка», «Значение», «Логин», … — произвольное), `kind` (подсказка отображения:
  `text` / `secret` / `url`), `ciphertext/iv/auth_tag`, `position`, timestamps.

Свойства:

- **Все значения полей шифруются** AES-256-GCM тем же мастер-ключом (включая описание и
  ссылку — требование «all-fields»). Plaintext остаются только структурные идентификаторы:
  `env_key` (нужен для связки с `secrets_item` и уникальности) и имена полей.
  Зашифрован и `title` — список карточек расшифровывается на сервере при рендере
  (масштаб владельца — сотни карточек, это дёшево).
- **Без обрезания длинных значений**: колонки `text`, валидация поднята до 256 КБ на поле
  (любой JWT/сертификат/ключ входит целиком; кап — только защита от злоупотребления).
- **Произвольные доп. поля**: строки `secrets_card_field` добавляются владельцем на карточку
  без миграций.
- **Связка с env-значением**: если у карточки есть `env_key`, поле «Значение» читает/пишет
  `secrets_item` этой комнаты (одно место истины, машинный API видит то же значение).
  Карточка без `env_key` (личный пароль) хранит все значения в своих полях.

### 3. Вход и 2FA (фаза 2)

Требование: один секрет + 2FA → полный доступ, не переспрашивать на каждую карточку.

- **TOTP (RFC 6238) — основной второй фактор.** Проверенная библиотека (`otplib`), не
  самописный крипто-код. СМС не делаем: платный шлюз + точка отказа в РФ (оценка из
  директивы разделяется); fallback — одноразовые recovery-коды (10 штук, в БД только хэши).
  Если владелец захочет СМС — добавим отдельной фазой.
- **Flow**: логин+пароль → (если TOTP включён) экран 6-значного кода → полная сессия
  (JWT-cookie как сейчас, с claim `mfa: true`). Дальше — полный доступ ко всем карточкам
  без переспросов до конца сессии.
- **Enrollment** в `/settings`: QR (`otpauth://`), TOTP-секрет в БД шифруется мастер-ключом.
  Пока 2FA не включён, `/secrets` доступен как сейчас (не ломаем себе прод), но GUI
  подсказывает включить; после включения — раздел требует `mfa: true`.
- **Той же фазой — hardening входа** (сейчас на `/api/auth/login` нет ни лимита, ни lockout):
  rate-limit (переиспользуем `lib/secrets/rate-limit`), блокировка после N неудач
  (username+IP, окно ~15 мин), audit-лог входов.

### 4. Mirror-API (ADR-0006) — УЖЕ ЭКСПОНИРОВАН

Механизм дублирования готов и живёт в проде с #54/#58: комната каждого проекта + rw-токен +
`POST /api/secrets` (upsert копии при создании/ротации секрета) + rate-limit + audit.
Контракт клиента — `docs/secrets-client-guide.md` (Node/Python/curl сниппеты).
Брейну можно запускать rollout client-рецепта по проектам. Второй слой квот
(per-project суточная квота поверх per-credential) — добавим, если появится реальный шум.

Карман сам себя не зеркалирует (его мастер-ключ — корень доверия, #008); прод-истина
секретов — `/etc/<app>/<app>.env`, Карман — зеркало+метаданные (ADR-0006).

### 5. Импорт (фаза 3)

CSV-импорт в выбранную комнату: загрузка файла → маппинг колонок на поля карточки →
создание карточек. Первым — формат экспорта браузера (url/username/password); экспорт
других менеджеров — по факту, формат уточняется с владельцем.

### 6. Шифрованный бэкап vault (фаза 4)

Дамп таблиц `secrets_*` + gpg-шифрование (паттерн R17 — offsite-бэкап ПДн), выгрузка вне
бокса. Мастер-ключ в бэкап не входит (хранится отдельно у владельца) — дамп бесполезен
без него даже в plaintext-виде, gpg — второй слой.

## Фазы реализации

| Фаза | Что | Статус |
|---|---|---|
| Ф0 | Ячейки + env-секреты + токены + машинный API + GUI | **в проде** (#48/#54/#58) |
| Ф1 | Карточки: миграция 0004, сервис, GUI карточек с произвольными полями | **реализовано** (PR feat/secrets-cards) |
| Ф2 | TOTP 2FA + recovery-коды + hardening логина (rate-limit, lockout, audit) | **реализовано** (PR feat/totp-2fa, миграция 0005) |
| Ф3 | CSV-импорт | **реализовано** (PR feat/secrets-csv-import) |
| Ф4 | Шифрованный gpg-бэкап vault + media/ (Яндекс.Диск) + наполнение личной ячейки | **механизм готов** (PR feat/vault-backup, `scripts/backup_vault.sh`), ждёт кред владельца |

Порядок Ф1↔Ф2 обсуждаем: Ф2 раньше повышает защиту до появления личных паролей в БД —
личную ячейку наполнять после Ф2.

## Открытые вопросы владельцу

1. **2FA**: согласен на TOTP-приложение (Google Authenticator / Aegis / …) как основной
   фактор + recovery-коды, без СМС? (рекомендация — да)
2. Слаг личной ячейки (`lichnoe`?) и момент наполнения (после Ф2).
3. Какие экспорты нужны импорту кроме CSV браузера (1Password, KeePass, …)?
